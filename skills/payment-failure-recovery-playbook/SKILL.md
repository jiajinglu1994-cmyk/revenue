# Payment Failure Recovery Playbook

## Purpose

这个 skill 解决的是“用户本来可能会续，但因为支付失败流失”的问题。

它通常比价格优化更快见效，也更适合先做止血。

## Key Concepts

### 1. 非自愿流失和主动取消要分开

支付失败恢复处理的是：

- 卡失效
- 余额不足
- 风控拒绝
- 支付渠道异常

不是用户主动取消。

### 2. 恢复动作通常由 3 层组成

1. 系统自动重试
2. 用户提醒与引导
3. 支付方式迁移

### 3. 恢复率之外还要看体验

同时关注：

- 恢复率
- 用户投诉
- 重试失败后的流失
- 支付方式切换成功率

## Required Inputs

- 支付失败率
- 失败原因拆分
- 当前重试节奏
- 当前提醒渠道
- 当前恢复率
- 支付方式分布

## Application

### Step 1: 先拆失败原因

至少分：

- 临时失败
- 永久失败
- 渠道故障
- 用户需手动更新支付方式

### Step 2: 设计重试策略

至少明确：

- 何时重试
- 重试几次
- 什么原因不重试

### Step 3: 设计提醒与迁移

常见动作：

- 账单失败提醒
- 更新支付方式引导
- 切换支付方式入口
- 人工客服兜底

### Step 4: 定义恢复成功

不仅看支付成功，还看：

- 用户是否保留权益
- 是否造成额外退款或投诉
- 恢复后的后续续费表现

## Output Format

1. 失败原因结构
2. 重试策略
3. 提醒与迁移策略
4. 恢复率目标
5. 风险与护栏

## Examples

### 示例：海外信用卡失败恢复

优先动作可能不是降价，而是：

- 拆卡失效与余额不足占比
- 优化 D0 / D1 / D3 重试节奏
- 强化更新卡片入口

## Common Pitfalls

- 不拆失败原因
- 所有失败都按同一节奏重试
- 没有支付方式迁移入口
- 只看恢复率，不看投诉和退款
