# Commercial Prioritization Advisor

## Purpose

商业化需求的排序不能只看“老板最急”或“预计收入最高”。这个 skill 用来帮你选对排序逻辑，并把需求放到统一评价口径里。

## Key Concepts

### 1. 不同需求，不能硬用同一把尺子

例如：

- 定价类更看弹性和风险
- 渠道类更看回本和规模化
- 留存类更看覆盖人群和长期价值
- 基建类更看解锁能力和复用价值

### 2. 先确定排序框架，再给结果

常见框架包括：

- 收入 / 利润 / 复杂度打分
- 回本优先
- 风险优先
- 战略门槛优先
- 止血优先

### 3. 商业产品优先级必须有“不做”选项

如果一个需求：

- 收益路径说不清
- 风险高于收益
- 当前组织承接不了
- 没法验证

就应该明确放进“不做”或“暂缓”。

## Recommended Criteria

默认使用下面 6 个维度，按场景做增减：

1. 收入影响
2. 毛利或利润影响
3. 回本速度
4. 验证确定性
5. 战略必要性
6. 实施复杂度

## When to Use

- 一次要排多个商业化需求
- 收入诉求和体验诉求冲突
- 领导层、运营、研发对优先级意见不一致
- 需要建立持续可复用的商业化排序框架

## Application

### Step 1: 先识别需求类型

商业产品常见需求类型：

- 定价与套餐
- 拉新转化
- 续费与复购
- 召回与挽回
- 商业化基建
- 渠道与投放效率

### Step 2: 给每类需求选主评估逻辑

- 定价类：收入弹性 + 风险
- 渠道类：回本 + 规模化
- 续费类：留存改善 + 人群覆盖
- 基建类：解锁能力 + 后续复用价值
- 功能类：收益路径 + 验证难度

### Step 3: 判断需要哪种排序框架

常见选择：

- 收入/利润/复杂度打分
- 回本优先
- 风险优先
- 战略门槛优先
- 分层路线图，先止血再增长

不要强行用单一分数覆盖所有需求。

### Step 4: 输出优先级建议

输出必须包含：

- P0：必须立即做
- P1：应在本季度推进
- P2：可作为验证或储备
- 不做：当前不值得投入

## Output Format

1. 推荐排序逻辑
2. 适用原因
3. 需求分组结果
4. 每组的核心判断标准
5. 需要补的数据或实验

## Examples

### 示例：季度商业化需求池排序

候选项：

- 年卡涨价灰度
- 支付失败恢复
- 海外新渠道试投
- 高阶权益包页改版

合理输出不是简单排 1 到 4，而是：

- P0：支付失败恢复
- P1：年卡涨价灰度
- P2：高阶权益包页改版
- 限额试验：海外新渠道试投

原因是这四类需求的收益路径、风险和验证方式不同。

## Common Pitfalls

- 只按预计收入排，忽略利润和风险
- 用同一把尺子衡量完全不同类型的需求
- 不区分立即止血和长期建设
- 把缺数据需求默认排后，而不是先设计验证
