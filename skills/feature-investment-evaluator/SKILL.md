# Feature Investment Evaluator

## Purpose

用商业化视角评估一个功能值不值得做，避免“看起来重要”但实际上不赚钱、不提留存、还增加复杂度的需求挤占资源。

## Use This When

- 需求和收入、留存、复购、召回相关
- 功能成本不低，至少需要一个完整迭代
- 需要向领导或跨团队解释为什么做或为什么不做
- 需要在几个候选商业化功能之间取舍

## Inputs

- 功能说明与目标用户
- 预期作用环节：转化、留存、复购、召回、客单价、抽佣
- 开发成本与维护成本
- 可能的运营支持成本
- 可能的技术风险和依赖
- 如果不做，会损失什么

## Evaluation Workflow

### Step 1: 识别价值来源

功能的价值来源通常只有 5 类：

1. 拉高首购或首付费转化
2. 提高续费、复购或召回
3. 拉高客单价、ARPU 或抽佣
4. 降低退款、作弊、履约损失
5. 提供战略必要性，例如大客户门槛、关键合规、平台能力

### Step 2: 拆解收益路径

收益不能只写“提升收入”，要明确：

- 作用在哪个漏斗环节
- 作用的人群有多大
- 作用强度怎么估
- 见效周期多久

### Step 3: 拆解成本

至少拆 4 类成本：

- 一次性研发成本
- 持续维护成本
- 运营成本
- 复杂度成本

复杂度成本包括：埋点、配置、客服解释、内容生产、对既有链路的扰动。

### Step 4: 判断验证难度

优先做可验证的功能，而不是只能“凭感觉会更好”的功能。

输出时明确：

- 能否 A/B test
- 能否分人群灰度
- 最短验证周期
- 核心成功指标

### Step 5: 输出建议

只允许输出以下建议之一：

- 现在做
- 先小实验
- 作为配套能力排期
- 不做

## Output Format

1. 功能价值摘要
2. 收益路径
3. 成本结构
4. 验证路径
5. 结论和优先级
6. 风险和依赖

## Common Pitfalls

- 把“战略价值”当成免算账理由
- 只估收益，不估运营负担
- 忽略与现有套餐和策略的冲突
- 没有实验路径就直接全量开发
